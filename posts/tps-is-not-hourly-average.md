PG 연동사에서 "취소 노티를 3~5초 지연 전송해달라"는 요청이 왔습니다. 중복 취소 처리를 막기 위해서라는데, 근거는 "서버 부담"이었습니다. 주장을 검증하려면 먼저 실제 부하를 측정해야 했습니다.

## 시간 평균의 함정

가장 흔한 실수는 시간당 건수를 3600으로 나눠 "TPS"라고 부르는 것입니다. TPS는 **초 단위 동시 처리량**입니다. 1시간 평균으로 내는 순간 피크가 평탄화되어 실제 부하를 과소평가하게 됩니다. 점심시간 10분에 몰린 트래픽이 한 시간 평균에 녹아 사라지는 식입니다.

## 피크는 MAX가 아니라 p99

그렇다고 초 단위 최댓값(MAX)을 피크로 잡는 것도 틀렸습니다. MAX는 outlier입니다. 시스템이 **상시 견뎌야 할 현실적인 피크는 분포의 p99**입니다.

측정은 3단계로 진행했습니다.

1. **시간대 좁히기** — 시간 단위 집계로 트래픽이 몰리는 구간을 찾는다.
2. **초 단위 실측** — 해당 구간을 초 단위로 집계한다.
3. **p99 확정** — 초당 처리량 분포에서 p99를 피크로 잡는다.

승인과 환불을 한 쿼리로 분리 집계할 때는 boolean 집계가 깔끔합니다.

```sql
SELECT
  DATE_FORMAT(created_at, '%H:%i:%s') AS sec,
  SUM(payment_type = 1 AND state = 'C') AS approve_cnt,
  SUM(payment_type = 2 AND state = 'C') AS refund_cnt
FROM payment
WHERE created_at BETWEEN :from AND :to
GROUP BY sec;
```

## 측정이 협상의 프레임을 바꾼다

측정 결과 피크는 **1~2 TPS**였습니다. "서버 부담"이라는 명분은 그 자리에서 소멸했습니다.

하지만 여기서 "부하 없으니 요청 거절"로 가면 협상이 막힙니다. 부하 명분이 사라졌으니, 프레임을 **설계 적절성과 책임 분담**으로 옮겼습니다.

- "3~5초 지연"은 근본 해결이 아니라 확률적 완화책입니다. 동기 응답이 지연 시간을 초과하면 중복은 재발합니다. 근본 해결은 수신 측의 **멱등(idempotency) 처리**입니다.
- 그래서 A안(자기 요청에 대한 취소 노티 억제로 레이스 컨디션을 구조적으로 제거)을 유도하고, 불가하면 B안(연동사별 설정 기반 지연)을 fallback으로 제시했습니다.

덧붙여, 지연을 구현해야 한다면 `Thread.sleep`은 금물입니다. 컨슈머 스레드(concurrency=2)를 묶어 처리량이 `concurrency / 지연시간`으로 급락하고 Kafka 리밸런싱이 폭주할 수 있습니다. 지연은 Redis ZSet이나 DB 예약 같은 **deferred 발송**으로 구현해야 지연 시간이 처리량과 무관해집니다.

측정 없이는 반박도 설계도 없습니다. 쿼리 한 벌이 회의 몇 번보다 강했습니다.
