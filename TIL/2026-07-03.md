## 배운 것

| # | 내용 |
|---|------|
| 1 | `Merchant` 엔티티의 `deleteMerchantStatus()`는 status만 변경하고 `delete_time`은 백오피스에서 세팅하지 않음 — `delete_time`은 타 시스템(결제 BE)에서 채우므로 삭제 여부의 원천은 `delete_time` 컬럼 |
| 2 | SQL에서 `column != 'DELETED'` 조건은 `NULL` 값 행도 함께 제외함 (`NULL != 'X'`는 UNKNOWN) — 의도치 않은 데이터 누락 원인 |
| 3 | UTC 저장 DB에서 KST 기준 기간 필터링 시, 컬럼에 `CONVERT_TZ`를 씌우지 말고 KST 경계를 UTC로 환산한 범위 조건(`create_time >= '2025-12-31 15:00'`)을 쓰면 인덱스를 탈 수 있음 |
| 4 | 코드베이스 컨벤션: Agency/PaymentCompany/Terminal 등 다른 도메인은 모두 `deleteTime.isNull()`을 삭제 필터로 사용 — 통계만 status 기반이라 예외였음 |
| 5 | 차트 컴포넌트 재사용 패턴: 데이터 형태(MonthlyStatItem)는 유지하고 표시 관련 prop(`barName`, `xTickFormatter`)에 기본값을 주면 기존 사용처 무변경으로 연도별 차트까지 커버 가능 |
| 6 | 연도별 집계는 특정 연도 선택 개념이 없으므로 WHERE 연도 범위 없이 전체 기간 GROUP BY — 누적 꺾은선의 마지막 값이 곧 "현재까지 총 가맹점 수"가 됨 |
| 7 | `MerchantFile`은 `@SQLDelete`(repository.delete() 시 soft-delete)지만 `@Where`가 없어 조회 시 삭제행도 반환 → 목록 쿼리에 `delete_time IS NULL` 필수. 삭제자/사유를 함께 남기려면 delete() 대신 dirty-checking으로 `updateDeleteTime()`+`updateMessage()` |
| 8 | `ApiService.call`은 응답 Content-Type이 **정확히 `application/zip`일 때만 Blob** 반환(그 외 JSON 파싱) → 단건 원본 다운로드도 BE가 `application/zip` 헤더로 응답해야 함(저장 파일명/확장자는 FE `a.download`가 원본명으로 강제) |
| 9 | `ApiService.multipart`는 `files[]`+`data`(JSON blob)만 전송하고 경로변수 폼필드를 안 붙임 → 업로드 API는 **정적 URL**로 두고 merchantId를 `data`에 담아야 함(`{{merchantId}}` 치환 불가) |
| 10 | SFTP 물리삭제는 트랜잭션 밖 → 다건 삭제는 "물리 rm 성공분만 DB 반영 + 건별 결과 반환" best-effort, 개별 rm 실패는 **재-throw 금지**(전체 롤백 방지). rm→DB 순서(잔여 물리파일이 전체 다운로드에 노출되는 것 방지) |
| 11 | BE 권한 판정은 FE와 동일하게 **`realm_access.roles`** 기준으로 읽어야 정확(JwtConverter는 authorities를 `resource_access`로 매핑하므로 소스가 다를 수 있음) |
| 12 | 빌드 실패를 삼키는 안티패턴: `build.sh`에 `set -e`가 없으면 `next build` 실패 후 후속 스텝 성공 시 exit 0 + Jenkins "Publish over SSH"가 결과를 SUCCESS로 승격 → 실패가 성공처럼 보임. `No build cache`/telemetry 문구는 무관 |
| 13 | 응답 DTO 필드명 변경은 "새 API가 없어도" 계약 변경 — 같은 엔드포인트라도 FE/BE 스키마 불일치 시 신규 필드가 빈값. 변경 없는 시리즈(접수)만 동작하는 증상으로 BE 미배포를 식별 가능 |
| 14 | 엔티티 직접 수정 금지(dqms-persistence-lib) 제약에서 다중 테이블 상태 변경은 QueryDSL bulk update(`update().set().where().execute()`)로 세터 없이 처리 + 단일 `@Transactional`로 원자성 확보 |
| 15 | 월별 누적이 매년 0으로 리셋되어 전체와 불일치하던 문제는 "기초 잔액(이전연도까지 누적)"을 시드로 주면 전 기간 정합 — 연도별은 전 기간 GROUP BY라 불필요 |
| 16 | 폐점 연도 GROUP BY에 `baseTime.isNotNull()`을 넣어 delete_time 없는 DELETED 행이 "연도 미상" 버킷으로 새는 것 방지 |
