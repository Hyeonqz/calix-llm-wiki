| 2 | SSOT는 "한 곳에 정의"만으론 부족하고, 파생값을 소비하는 다른 경로(SQL CASE 리터럴)가 SSOT를 **참조**하도록 바꿔야 진짜 드리프트가 막힌다. 참조가 불가능한 경계는 정합성 테스트로 봉인 |
| 3 | 금융 배치는 "매칭 실패 시 fallback"보다 "fail-loud"가 옳다 — 조용히 틀린 값보다 감지 가능한 유실이 낫다. 단, 단건은 즉시 실패, 배치(range)는 전량 처리 후 일괄 보고가 운영 편의상 유리 |
| 4 | QueryDSL 중복 제거: projection은 `ConstructorExpression` 반환 헬퍼로, join `on`/`where` 조건은 `BooleanBuilder`(=Predicate) 반환 헬퍼로 빼면 타입 안전하게 재사용된다. 단 파생 쿼리별 도메인 차이(QBANG 제외 유무 등)는 공통 헬퍼 적용 범위를 반드시 한정해야 한다 |
| 5 | 컴파일 검증이 불가능한 환경(스냅샷 선행 블로커)에선 오버로드된 메서드에 메서드 레퍼런스(`this::foo`)를 거는 것 자체를 피해 이름을 분리하는 게 안전 — 정적 안전성을 컴파일러에 의존할 수 없을 때는 모호성의 여지 자체를 없앤다 |
| 6 | 배치 이력(진행/성공/오류) 처리 골격이 여러 서비스에 반복될 때, 조회·모수·로그태그만 파라미터(`Supplier`/tag)로 받는 템플릿 메서드로 위임시키면 중복이 사라진다. 단 트랜잭션 경계가 걸린 골격의 상위(BatchService) 승격은 `@Transactional` 위치 확인 후 별도 단계로 |
| 7 | 구현과 리뷰는 분리된 패스로 — 같은 맥락에서 자가승인하지 않고 code-reviewer에게 리팩토링 관점만 따로 맡기니 중복·드리프트·네이밍 개선점이 체계적으로 드러났다 |
| 8 | 폐쇄망 doc-driven 이식 문서는 스니펫이 아니라 **디스크 최종본 전체**를 담아야 붙여넣기 불일치가 없다. 티켓(추적용 요약)과 이식용 문서(전체 소스)는 역할을 분리하고 상호 링크 |
