## 고정폭 파일 빌더에서 LinkedHashMap 삽입 순서가 곧 출력 컬럼 순서 — 필드 하나 옮기면 4곳을 동시에 고쳐야 한다
태그: #java #architecture

크레소티 매입반송 대사 파일은 각 레코드를 DTO의 `toMap()`이 만드는 `LinkedHashMap`으로 표현하고, `String.join(delimiter, map.values())`로 라인을 만든다. 즉 **`map.put(...)` 호출 순서 = 파일 컬럼 순서**다. `HashMap`이면 순서가 깨져 파일이 어긋나지만, `LinkedHashMap`이라 삽입 순서가 그대로 고정폭 레이아웃이 된다.

함정은 필드 하나를 추가/삭제할 때다. ① 필드 선언, ② `toMap()`의 put 순서, ③ `of()` 팩토리의 `setValue`, ④ 호출부의 인자 전달 — 이 4곳을 **모두 일관되게** 고쳐야 한다. 하나라도 누락되면 그 지점 이후 모든 컬럼이 한 칸씩 밀려 상대방 파서가 전부 깨진다(컴파일은 통과하므로 조용히 사고난다).

```java
public LinkedHashMap<String, String> toMap() {
    var map = new LinkedHashMap<String, String>();
    map.put("lineType", ...);      // 순서 = 컬럼 위치
    map.put("recordType", ...);
    // ... 여기 순서를 바꾸면 파일 레이아웃이 바뀐다
    return map;
}
// MapUtils: String.join(DELIMITER, map.values()) + LF
```

## 필드명은 "명(name)"인데 값은 코드가 담긴다 — 이름-값 불일치는 컴파일러가 못 잡는다
태그: #java

같은 파일의 `DataHeader.of()`에서 `paymentCompanyName`(간편결제사"명") 필드에 `PaymentCompanyCode.getCodeByName(brand)` 결과를 넣고 있었다. 그런데 이 메서드는 이름이 아니라 **2자리 코드("UP","WP")를 반환**한다(`enum("UnionPay","UP")`에서 name으로 조회해 code를 돌려줌). 결국 "명" 필드에 실제로는 코드가 저장된다.

둘 다 `String`이라 컴파일러는 못 잡고, 파일도 정상 생성되고, 길이(AN 20)도 안 넘쳐 아무 에러가 없다. 스펙 변경으로 이 필드를 다른 레코드로 옮기려는데, 새 위치에 넣을 값이 "기존과 같은 코드"인지 "실제 페이사명"인지 확정하지 않으면 **의미가 바뀐 채로 그대로 이식**될 뻔했다.

교훈: 필드명(`xxxName`)과 값 소스(`getCodeByXxx`)가 어긋나는 지점은 리팩터링/이관 전에 반드시 값의 의미를 재확인해야 한다. 어제(`representativePhone`에 대표자명이 들어가던 복붙 버그)와 같은 계열 — **타입만 맞으면 잘못된 값도 성공한다.**

```java
// enum: name="UnionPay", code="UP"
public static String getCodeByName(String name) { ... return code; } // "UP" 반환
header.paymentCompanyName.setValue(getCodeByName(brand)); // "명" 필드인데 코드가 들어감
```

## 요청/응답 DTO에서 필드 하나를 지우면 그 값을 만들던 상류 파이프라인이 통째로 죽는다 — 숨은 안티패턴까지 청소된다
태그: #refactoring #architecture

모레타페이 승인/취소 요청의 `merchantInformation`에서 `merchantPhone`을 뺐을 뿐인데, 그 필드 하나가 물고 있던 체인이 전부 불필요해졌다:

- QueryDSL projection의 `Q_MERCHANT.mainPhone` 조회 컬럼
- 그 암호문을 풀던 `vaultService.decryptWithDekByEncryptedText(...)` 호출
- 그 때문에 존재하던 `VaultService` 의존(필드·생성자 파라미터)
- 복호화 결과를 담던 record 필드와 `decryptedMerchantSummary` 팩토리

핵심은 **"출력에서 안 쓰는 필드"를 지우면 그 값을 생산하던 상류 전부가 dead code로 확정된다**는 것. 리팩터링을 wire(요청 스펙) 끝에서 시작해 역방향으로 따라가면 어디까지 지울 수 있는지 자연히 드러난다. 덤으로 아래 두 안티패턴(트랜잭션 내 원격 복호화, PII 로그 노출)도 필드가 사라지며 같이 청소됐다 — 의도한 게 아니라 필드 제거의 부수효과였다.

## readOnly 트랜잭션 안에서 Vault 복호화를 부르면, DB 커넥션을 쥔 채 외부 HTTP 왕복을 한다
태그: #transaction #payment #jpa

승인 플로우는 결제사(PSP) HTTP 호출을 `@TransactionalEventListener(AFTER_COMMIT)` + `@Async`로 트랜잭션 밖에 잘 빼놨다. 그런데 정작 그 직전 **데이터 준비 단계**에 결합이 숨어 있었다:

```java
@Transactional(readOnly = true)          // 트랜잭션 열림
public Optional<AppMerchantSummary> findMerchantSummaryByMerchantId(Long id) {
    var row = jpaQueryFactory.select(...).fetchOne();          // DB 커넥션 점유
    var phone = vaultService.decryptWithDekByEncryptedText(row.mainPhone()); // ← Vault로 HTTP 왕복
    ...                                                        // 커넥션 잡은 채 외부 I/O
}
```

Vault transit 복호화는 Vault 서버로의 네트워크 왕복이다. 즉 트랜잭션(=커넥션)이 열린 채 외부 호출이 일어나, Vault 지연 시 **DB 커넥션 풀이 고갈**된다. "외부호출-트랜잭션 분리"를 점검할 때 눈에 띄는 PSP 호출만 보면 안 되고, 조회·복호화 같은 **준비 단계의 원격 호출**도 트랜잭션 경계 안인지 봐야 한다. 원칙: 트랜잭션 안에서는 암호문만 조회해 반환하고, 복호화는 트랜잭션 밖(서비스 계층)에서.

## 결제에서 타임아웃은 "실패(FAILED)"가 아니라 "불확정(UNKNOWN)"이다 — 무조건 실패확정은 승인 유실을 만든다
태그: #payment #consistency

승인 API가 `catch (Exception)`에서 타임아웃/네트워크 예외까지 전부 원장을 `FAILED`로 확정하고 있었다. 문제 시나리오: PSP는 **실제로 승인(돈이 이동)** 했는데 읽기 타임아웃(60s)이 나거나 응답 수신 후 영속화 직전 크래시가 나면 → 내부 원장은 `FAILED`. **PSP=승인 / 내부=실패**의 정합성 불일치가 남고, 이걸 되돌릴 조회(inquiry)·재대사·망취소(reversal) 경로가 없으면 그대로 굳는다. 결제에서 가장 비싼 실패 유형(승인 유실).

교훈: 타임아웃은 사실을 "실패"로 단정할 근거가 아니다. (1) 호출 직전 `승인요청` 중간상태 기록, (2) 타임아웃/네트워크 예외는 `FAILED`가 아닌 `UNKNOWN/보류`로 전이, (3) 스케줄러가 PSP 거래조회로 확정 or 망취소. 최소한 **명시적 거절과 불확정(타임아웃)을 예외 타입으로 구분**해야 한다. 이런 상태 모델 결정은 ADR로 남길 값어치가 있다.

## 지시받은 getter가 코드에 없으면 그대로 쓰지 말고 grep으로 실존을 확인한다 — 인접 매핑과의 일관성으로 교차검증
태그: #discipline

"`merchant.getBusinessCategoryCode()`로 매핑해줘"라는 요청을 받았지만, `apps/core`·전체 모듈 grep 결과 `businessCategoryCode` 심볼이 **하나도 없었다.** 실존하는 건 `businessCategory` 필드(Lombok getter `getBusinessCategory()`)뿐. 게다가 취소(5.3) 쪽은 이미 `merchant.getBusinessCategory()` → JSON `merchantCategoryCode`로 매핑하고 있어, 승인 쪽도 같은 소스로 맞추는 게 일관됐다.

지시의 심볼명을 액면 그대로 타이핑했으면 컴파일 에러가 났을 것이다. **요청에 등장한 API가 실재하는지 먼저 확인하고, 오타로 보이면 인접한 동종 매핑과의 일관성으로 정답을 역추론**한 뒤, 무엇을 왜 바꿔 적용했는지 사용자에게 명시한다. 어제(`representativePhone`에 대표자명이 들어가던 복붙 버그)와 같은 계열 — 타입만 맞으면 잘못된 값·없는 값도 조용히 지나간다.
