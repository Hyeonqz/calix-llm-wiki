## 배운 것

| # | 내용 |
|---|------|
| 1 | 정산/집계 쿼리에서 "N건 중 1건만 누락"은 대개 **INNER JOIN 조인 파트너 부재**가 원인 — payment는 멀쩡한데 파생 조인 키에서만 탈락하면 에러 없이 사라짐 |
| 2 | 결제사(brand)와 수수료 정책(영세/일반)을 `pymt_cmpy` **문자열 하나에 인코딩**(`Alipay+` vs `Alipay+_Micro`)하면, 쿼리가 그 인코딩을 정확히 재현해야만 매칭됨 → 깨지기 쉬운 "인코딩된 키" 안티패턴 |
| 3 | `CASE WHEN merchant_class = MICRO ...`에서 `merchant_class`가 **NULL이면 조건이 false**로 빠져 자동으로 일반(GENERAL) 취급 → NULL이 조용한 유실을 유발 |
| 4 | 디버깅 정석: INNER JOIN을 LEFT JOIN으로 바꿔 그 1건만 조회하면 **어느 조인 컬럼이 NULL로 나오는지 = 탈락 지점**을 바로 특정 가능 |
| 5 | 금융 배치에서 매칭 실패 시 base 요율로 **fallback 금지** — 영세/일반 요율이 달라 조용히 틀린 금액을 정산함. "틀리느니 멈춘다"가 원칙 |
| 6 | 유실 안전망은 **"정산 모수(COMPLETED·QBANG 제외) − 실제 정산집합"** 차집합으로 구현하면 원인 불문(요율행 부재·시간창·키 불일치) 모든 누락을 한 번에 감지 가능 |
| 7 | 파생 규칙 드리프트 방지: SQL CASE(DB 실행 필요)는 남기되, **테스트된 Java SSOT 함수** + brand×class 유한 조합 전수 **정합성 테스트**로 둘이 어긋나지 않게 고정 |
