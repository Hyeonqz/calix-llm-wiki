## 캐시 TTL이 유효시간 정책을 대신하고 있으면, 그 정책은 코드 어디에도 안 적혀 있다
태그: #architecture

QR 결제 검증이 Redis `qrCode:{payload} → transactionNo` 매핑을 조회해 "있으면 유효, 없으면 무효"로 판정하고 있었다. 겉보기엔 **매핑 저장소**지만, TTL이 120초라 실제로는 **"QR은 2분간만 유효하다"는 만료 정책**을 수행하고 있었다. 코드에는 시간 검증이 한 줄도 없다.

이게 위험한 이유는 정책이 **명시되지 않은 채 인프라 설정값에 숨는다**는 것. 요구사항을 확인해보니 기업 QR은 생성 2~3시간 뒤에도 결제될 수 있어야 했다. 즉 `Duration.ofSeconds(120)` 상수 하나가 **아무도 모르게 정상 거래를 죽이고 있었다** — 그것도 "Redis에 없음 = 유효하지 않은 QR"이라는, 원인을 숨기는 에러 메시지로.

교훈: 캐시 미스를 **비즈니스 판정의 근거**로 쓰면 안 된다. 캐시 미스는 "모른다"이지 "무효다"가 아니다. Redis 재기동·eviction에도 같은 이유로 정상 거래가 전부 무효가 된다. 판정 근거는 원장(RDB)에 있어야 하고, 만료는 상태(state) 또는 명시적 시간 조건으로 코드에 드러나야 한다.

## 검증의 "질문"을 바꾸면 인프라가 통째로 사라지기도 한다
태그: #architecture

위 Redis를 어떻게 대체할지 고민하다가, **질문 자체가 틀렸다**는 걸 깨달았다. 코드가 묻고 있던 건 *"이 payload가 Redis에 있나"* 였는데, 진짜 물어야 할 건 *"이 거래번호가 우리가 발급한 실제 거래이고, 아직 결제 가능한 상태인가"* 였다.

두 번째 질문의 답은 이미 **결제 원장에 100% 있다.** 게다가 QR payload를 뜯어보니 거래번호가 그 안에 박혀 있었다(`.../endpoints/app/{거래번호 20자}{EMVCo TLV...}`). 즉 payload→거래번호 매핑을 저장할 이유가 애초에 없었다 — **파싱하면 된다.**

결과적으로 Redis 의존이 사라지면서 (1) POS가 발급한 QR(매핑이 안 심기던 경로)도 자동으로 통과하고, (2) 기존 TTL 버그도 같이 사라지고, (3) Redis 장애 시 결제 실패하던 취약점도 없어졌다. **기능 추가가 아니라 잘못된 질문을 걷어낸 것뿐인데 세 문제가 동시에 풀렸다.**

## 앞부분이 동일한 긴 문자열 컬럼은 prefix index가 무의미하다 — 인덱스 조회 후 "이미 찾은 행"과 대조하라
태그: #database

payload로 거래를 찾는 방법이 둘이었다.

- **A**: payload에서 거래번호를 파싱 → `payment.transaction_no`(유니크 인덱스) 조회. 빠르지만 **payload를 그대로 신뢰**한다.
- **B**: 발급 이력 테이블의 `qr_code`(varchar 1024) 컬럼으로 역조회. 위조는 못 하지만 **인덱스가 없다.**

B에 prefix index를 걸면 되지 않나 싶었는데, **우리 payload는 앞부분이 `https://api.qr-road.com/endpoints/app/`로 전부 동일하다.** prefix index의 선택도가 사실상 0이라 풀스캔과 다를 게 없다. 승인 핫패스에서는 못 쓴다.

답은 **둘 중 하나를 고르는 게 아니었다.** A로 인덱스 조회해서 행을 찾은 뒤, **그 행의 `qr_code`를 꺼내 요청 payload와 문자열 비교**하면 된다. `qr_code`를 **검색 조건이 아니라 비교 대상**으로만 쓰므로 인덱스가 필요 없다. B의 비용은 안 내고 B의 안전성만 가져온다.

이 대조가 필요했던 이유는 응답에 **가맹점 대표자명·전화번호·계좌번호가 복호화되어 실리기** 때문. 거래번호만 위조하면 민감정보가 나간다.

## `assert`는 런타임 기본 비활성이라, null 가드로 쓰면 그 자리에서 NPE가 난다
태그: #java

Kafka 리스너에 이런 패턴이 있었다.

```java
try {
    message = objectMapper.readValue(record.value(), Msg.class);
} catch (JsonProcessingException e) {
    log.error("Json Parse Error : {}", e.getMessage(), e);   // message = null 인 채로 진행
}
try {
    assert message != null;        // JVM 기본(-ea 없음)에서는 아무 일도 안 함
    process(message);              // → NPE
} catch (Exception e) {
    log.error(e.getMessage(), e);  // 응답도 ack도 없이 종료
}
```

`assert`는 `-ea` 플래그 없이는 **바이트코드가 실행되지 않는다.** 운영 JVM은 보통 안 켜므로 이 가드는 없는 것과 같다. 게다가 NPE가 광역 `catch (Exception)`에 먹히면서 **Kafka reply도 안 나가고 `acknowledge()`도 안 된다** → 요청한 쪽은 타임아웃, 오프셋은 미커밋.

파싱 실패는 **정상적인 실패 경로**다. 예외를 삼키고 계속 진행하지 말고, 그 자리에서 실패 응답 + ack 후 return 해야 한다.

## 응답을 보내고 `return`하는 `void` 검증 메서드는 호출부의 흐름을 못 멈춘다 — 응답이 두 번 나간다
태그: #java

```java
// 호출부
validConsumerRecordValue(record, message);   // 실패하면 실패 응답을 보내고 return
OqmsPaymentResponseMessage response = process(message);
replyHandler.sendReply(record, response);    // ← 검증 실패였어도 여기까지 온다
```

검증 메서드 내부의 `return`은 **자기 자신만 빠져나온다.** 호출부는 그걸 알 방법이 없어 그대로 다음 줄을 실행하고, 결국 **Kafka reply가 2회 전송된다.** 첫 응답은 실패, 두 번째는 (검증을 안 거친) 처리 결과다. 수신 측이 어느 걸 채택하는지에 따라 동작이 갈린다.

리뷰에서 자주 놓치는 이유는 메서드 안에 `return`이 **보이기** 때문. 부작용(응답 전송)과 흐름 제어를 한 메서드에 섞으면 이런 착시가 생긴다. 검증 결과는 반드시 **반환값으로** 호출부에 전달해야 한다.

```java
if (!validateAndReplyOnFailure(record, message)) {   // boolean 반환
    acknowledgment.acknowledge();
    return;
}
```

이름도 같이 고쳤다. `isValidXxx`는 순수 판정처럼 보이는데 내부에서 Kafka 응답을 보내므로 정직하지 않다 → `validateAndReplyOnFailure`.

## JPA의 inverse `@OneToOne`(mappedBy)은 기본 EAGER다 — "lazy라서 트랜잭션이 필요하다"는 전제가 틀릴 수 있다
태그: #jpa

`payment → paymentStep1Request → paymentStep1Response.qrCode`로 내려가는 조회를 짜면서, 설계 문서에 *"lazy 로딩된 연관에 접근하므로 `@Transactional` 안에서 수행한다"*고 적었다. **틀렸다.**

`@OneToOne(mappedBy = "payment")` 같은 **inverse(비소유) 측 OneToOne은 JPA 기본 전략이 EAGER다.** 소유 측은 `fetch = LAZY`를 지정할 수 있지만, inverse 측은 프록시를 만들려면 연관이 존재하는지 알아야 하고 그러려면 어차피 쿼리를 날려야 해서 하이버네이트가 즉시 로딩한다. 즉 `LazyInitializationException` 위험은 애초에 없었다.

실제로 문제가 되는 건 **반대 방향**이었다. `Payment`에는 inverse `@OneToOne`이 4개(`paymentStep1Request`, `paymentStep2Request`, `paymentResult`, `systemCancellation`) 달려 있어, `findByTransactionNo()` **한 번에 부가 SELECT가 여러 번** 나간다. 승인 핫패스에서는 `@EntityGraph`나 fetch join이 필요하다.

교훈: "lazy니까 트랜잭션"은 검증 없이 쓰기 쉬운 문장이다. **연관의 소유/비소유와 기본 fetch 전략을 실제로 확인하고 쓰자.**

## 차단(reject) 정책을 새로 넣을 때는, 오탐을 진단할 로그가 같이 들어가야 한다
태그: #logging

"요청 payload가 발급한 QR과 정확히 일치해야 통과"라는 조건을 1차 배포부터 **차단**으로 넣기로 했다. 그런데 코드 리뷰에서 잡힌 게 — 불일치 시 로그가 이랬다.

```java
log.debug("{} 의 발급된 QR 과 요청 payload 가 일치하지 않습니다.", transactionNo);
```

문제가 둘이다. (1) **양쪽 값이 없다** → 왜 다른지 알 수 없다. (2) `debug`라 **운영에서는 아예 안 남는다.**

이 조건은 상대 시스템이 payload에 URL 인코딩이나 공백을 덧붙이기만 해도 **정상 거래가 전량 거절**된다. 그 상황에서 원인 규명 수단이 0이면, 차단 정책은 안전장치가 아니라 시한폭탄이다. **"이 검증이 오탐일 때 나는 무엇을 보고 알 수 있나"에 답할 수 없으면 차단으로 넣으면 안 된다.**

```java
log.warn("{} 발급 QR 과 요청 payload 불일치. issuedQrLength={}, issuedQrPrefix={}, payloadLength={}, payloadPrefix={}",
    transactionNo, len(issued), truncate(issued), len(payload), truncate(payload));  // 앞 60자만 (원문 전체 노출 방지)
```
