# TIL - 2026-07-13 (월)

## Jakarta Bean Validation은 중첩 객체에 `@Valid`가 없으면 cascade하지 않아 검증 애노테이션이 조용히 죽는다
태그: #spring-boot

Static QR 결제에서 `totalAmount="500.00"` 이 `NumberFormatException`을 냈다. 그런데 해당 필드엔 이미 `@Pattern("^[0-9]+$")`이 붙어 있었다. `"500.00"`은 이 패턴을 명백히 위반하는데도 통과했다는 것 자체가 **검증이 아예 실행되지 않았다는 증거**였다.

원인은 컨트롤러의 `@RequestBody @Valid`가 **최상위 필드까지만** 검증한다는 점. 중첩 객체(record 포함)로 내려가려면 그 필드에 `@Valid`를 명시해야 한다. 없으면 내부의 모든 제약이 런타임에 평가되지 않는다.

무서운 건 **코드를 읽으면 검증이 있어 보이고 리뷰도 통과한다**는 것. "검증이 있다"는 착각이 "검증이 없다"보다 위험하다. 이 결함은 서비스 단위 테스트로는 절대 안 잡히고 `@WebMvcTest`로만 잡힌다.

```java
public record AppPaymentRequest(
    String transactionNo,
    @Valid                    // 이게 없으면 아래 StaticQr 내부의 @Pattern/@Size 가 전부 무력
    StaticQr staticQr
) {
    public record StaticQr(
        @Size(max = 3) String currency,
        @Pattern(regexp = "^[0-9]{1,13}(\\.[0-9]{1,2})?$") String totalAmount
    ) {}
}
```

## 결제 금액은 `RoundingMode.UNNECESSARY`로 반올림을 예외화한다 — 조용한 반올림은 대사에서 추적 불가능한 차액이 된다
태그: #java

`BigDecimal price = BigDecimal.valueOf(Long.parseLong(totalAmount))` 가 버그의 직접 원인이었다. `String → long → BigDecimal` 2단 변환인데, 목적지(`BigDecimal`)와 DB 컬럼(`DECIMAL(12,2)`)은 소수를 다루는 반면 **중간의 `long`만 정수 전용**이라 소수 표현 능력을 버린다. 이 단계는 존재할 이유가 없다.

허용 자릿수를 넘는 금액을 만났을 때 `HALF_UP`으로 반올림하고 싶은 유혹이 있는데, **결제 금액엔 절대 쓰면 안 된다.** 고객이 보낸 금액과 원장 금액이 달라지는 순간 대사 단계에서 원인 추적이 불가능한 차액으로만 뒤늦게 드러난다. **모르면 반올림하지 말고 거절하라.**

`scale` 검사도 문자열이 아니라 파싱된 값으로 해야 한다. `BigDecimal`은 leading zero를 유효숫자로 세지 않으므로 zero-padding(`"000000000500"`)도 정수부 3자리로 정확히 계산된다.

```java
BigDecimal amount = new BigDecimal(rawAmount);          // long 경유 제거
if (amount.signum() <= 0) throw ...;                    // 0원/음수
if (amount.precision() - amount.scale() > 10) throw ...; // DECIMAL(12,2) 정수부 10자리
try {
    return amount.setScale(2, RoundingMode.UNNECESSARY); // "500.005" -> ArithmeticException
} catch (ArithmeticException e) {                        // 반올림하지 않고 거절
    throw new QRBankAppException(REQUEST_VALIDATION_INVALID_FORMAT, e);
}
```

## `IllegalArgumentException → 400` 광역 핸들러는 서버 버그를 클라이언트 오류로 위장시켜 5xx 알람을 무력화한다
태그: #spring-boot

`NumberFormatException`은 `IllegalArgumentException`의 하위 클래스다. 그래서 전역 핸들러의 `@ExceptionHandler(IllegalArgumentException.class)`에 걸려 **HTTP 400 + JVM 예외 메시지**(`For input string: "500.00"`)가 외부 결제사에게 그대로 나갔다.

문제가 셋이다. (1) 이 API의 계약은 "항상 HTTP 200, body의 `status.code`로 성공/실패 구분"인데 400을 반환해 계약을 깼다. (2) 내부 구현이 외부 파트너에게 노출됐다. (3) 가장 위험한 것 — **우리 파싱 버그(5xx급)가 400(클라이언트 잘못)으로 재분류**됐다. 모니터링은 보통 5xx에만 알람을 걸기 때문에 **이 장애는 알람을 울리지 않는다.**

교훈: 광역 `IllegalArgumentException` 매핑은 의도된 클라이언트 오류와 미포착 프로그래밍 버그를 구분하지 못한다. 클라이언트 오류는 명시적 도메인 예외로 던지고, 나머지 런타임 예외는 500으로 떨어져 알람이 울리게 둬야 한다.

에러 코드 선택도 같은 맥락이다. 검증 실패를 전부 `FL92("비정상적인 요청")`로 뭉뚱그리면, 이게 인증 계열 코드(FL90 인증실패 / FL91 HMAC실패) 옆에 있어서 결제사는 "금액 형식이 틀렸다"와 "HMAC이 깨졌다"를 구분할 수 없다. 위반된 제약 종류로 나눠 매핑했다.

```java
private QBangAppStatus toAppStatus(ObjectError error) {
    // getCodes() 는 [Pattern.dto.field, ..., Pattern] 순 — 마지막이 애노테이션 이름
    String[] codes = error.getCodes();
    String constraint = (codes == null || codes.length == 0) ? "" : codes[codes.length - 1];
    return switch (constraint) {
        case "NotNull", "NotBlank", "NotEmpty" -> REQUEST_VALIDATION_MISSING_PARAMETER; // FL01
        case "Size", "Length", "Max", "Min"    -> REQUEST_VALIDATION_DATA_LENGTH_EXCEEDED; // FL02
        default                                 -> REQUEST_VALIDATION_INVALID_FORMAT;    // FL03
    };
}
```

## 버그를 고칠 때 입력 규격을 좁히면 기존 연동이 깨진다 — 리스크가 비대칭이면 넓은 쪽이 기본값
태그: #devops

금액 정규식을 처음엔 `^(0|[1-9][0-9]{0,9})(\.[0-9]{1,2})?$`로 짜서 leading zero를 막았다. 코드 리뷰에서 잡혔는데, **수정 전 코드는 `Long.parseLong("000000000500")` → `500`으로 정상 처리하고 있었다.** 결제 연동에서 금액 고정폭 zero-padding은 흔한 관례라, 이를 막는 건 버그를 고치면서 새 장애를 만드는 것이었다.

리스크가 비대칭이라는 게 핵심이다. **허용은 틀려도 손해가 없고**(결제사가 zero-padding을 안 써도 `"000000000500"`과 `"500"`은 동일하게 파싱), **거절은 틀리면 즉시 장애**다. 외부 규격서로 확인되기 전까지는 넓은 쪽이 기본값이어야 한다.

같은 이유로 `@NotBlank`도 붙이지 않았다. 이 DTO는 DPM/SPM 공용이라, `@Valid` cascade를 켜는 순간 빈 `staticQr`를 보내던 **기존 DPM 트래픽이 거절**된다. 필수값 검사는 SPM 경로 내부로 옮겼다. 검증을 되살리는 변경은 **그동안 조용히 통과하던 트래픽을 전부 다시 심사**한다는 뜻이므로, 무엇이 새로 거절되는지 먼저 세어봐야 한다.

## 시끄러운 로그를 지우는 기준은 "볼 일이 있나"가 아니라 "이게 유일한 증거인가"다
태그: #logging

TCP 패킷 hex 덤프(`02 30 30 23...`)가 운영 로그를 도배해서 지우려 했다. 처음엔 DEBUG로 내리려다, 전용 파일로 분리하는 설계까지 갔다가, 결국 **둘 다 버렸다.**

판단을 가른 건 두 사실이다. (1) 정상 파싱된 전문은 이미 `request_log` 테이블에 파싱 결과 + correlationId로 적재된다 → **정상 케이스 hex는 DB와 100% 중복이라 아무것도 더 알려주지 않는다.** (2) 파싱이 깨지면 `parse()`까지 못 가서 **DB에 아무것도 안 남는다** → 이때만 원본 바이트가 유일한 증거다.

즉 hex의 가치는 100% 실패 케이스에 있었다. 그래서 정상 경로 hex는 DEBUG 가드로 밀어넣고, **기존 `log.error` 3곳에 원본 hex를 붙였다.** 운영 로그는 평상시 hex 0줄이 되고, 사후 분석 능력은 오히려 좋아졌다.

tcpdump나 Actuator 런타임 토글도 검토했지만 둘 다 **"지금부터"만 본다.** "어제 그 단말이 왜 깨졌나"에는 답을 못 한다. 로그를 지울지 말지는 **재현 가능성**으로 갈린다.

```java
// 정상: DB에 있음 → 운영에서 안 찍음
if (log.isDebugEnabled()) log.debug("[Request Hex] {}", PacketHex.of(buf));

// 실패: DB에 아무것도 없음 → 항상 찍음
log.error("Invalid message: Missing ETX/CR/LF. hex={}", PacketHex.of(buf));
```

## 로그 한 줄 찍자고 `ByteBuf.copy()`를 뜨면 해제 책임이 생긴다 — absolute getter는 공짜다
태그: #netty

Netty 디코더가 hex를 찍으려고 이렇게 하고 있었다.

```java
ByteBuf copy = buf.copy();          // refCnt 1인 새 버퍼 — release() 필요!
byte[] bytes = new byte[copy.readableBytes()];
copy.readBytes(bytes);
log.info("[Request Hex] : {}", ...);   // 이게 전부. copy는 여기서만 쓰임
```

`copy()`는 **독립된 새 버퍼**를 만들어 refCnt 1로 시작한다. `release()`하지 않으면 풀드 버퍼가 풀로 반환되지 않는다. **로그 한 줄 때문에 전문마다 버퍼가 샜다.**

`getBytes(int, byte[])`(absolute getter)를 쓰면 복사 없이 읽는다. Netty 계약상 `readerIndex`/`writerIndex`/`refCnt`를 **전혀 건드리지 않으므로** 하류 파싱에 영향이 없고 해제 책임도 안 생긴다. `readBytes()`(relative)와 달리 인덱스가 안 움직이는 게 핵심.

같은 파일에서 `buf.slice(...).retain()`도 발견했다. `slice()`는 원본과 refCnt를 공유하는데 `retain()`으로 +1 해놓고 아무도 release하지 않았다. `MessageToMessageDecoder`는 입력 버퍼를 `finally`에서 **1회만** release한다 → **정상 전문마다 refCnt 1 잔류.** 스코프를 안 벗어나는 slice에 `retain()`은 애초에 불필요하다.

```java
byte[] bytes = new byte[buf.readableBytes()];
buf.getBytes(buf.readerIndex(), bytes);   // 인덱스/refCnt 불변, 해제 불필요
```

## `new String(byte[])`의 플랫폼 기본 charset은 LRC 같은 비문자 바이트를 소리 없이 파괴한다
태그: #java

고정길이 전문 응답을 만드는 코드다. body를 `ISO_8859_1`로 인코딩해놓고, 되돌릴 때는 charset을 안 줬다.

```java
byte[] bodyBytes = body.getBytes(CharsetUtil.ISO_8859_1);
byte lrc = makeLrc(lrcTarget);        // XOR 결과 → 0x80 이상 가능
...
return new String(wrappingResult);    // ← 플랫폼 기본 charset!
```

`ISO_8859_1`은 0x00~0xFF를 U+0000~U+00FF에 1:1 대응시키는 **바이트 보존 charset**이라 왕복이 무손실이다. 그런데 기본 charset이 UTF-8인 환경(리눅스 컨테이너 `LANG=*.UTF-8`, JDK 18+)에서는 `0x80` 이상 단독 바이트가 **유효한 UTF-8 시퀀스가 아니다.** → `U+FFFD`(replacement char)로 치환되고, 파이프라인의 `StringEncoder(ISO_8859_1)`가 이를 `?`(0x3F)로 내보낸다.

**결과: LRC가 깨져 단말이 응답을 거절한다.** LRC 값에 따라서만 터지므로 "가끔 결제가 안 된다"로 보인다. `byte[]` ↔ `String` 왕복에 charset을 안 주는 건 문자 데이터에서도 위험하지만, **바이너리 프로토콜에서는 즉시 데이터 파괴**다.

```java
return new String(wrappingResult, CharsetUtil.ISO_8859_1);
```

## logback의 `<springProfile>`은 `<appender>`/`<logger>`/`<root>` 안에 중첩할 수 없다 — 다만 WARN이라 기동은 된다
태그: #spring-boot

프로파일별로 appender를 다르게 붙이려고 `<logger>` 안에 `<springProfile>`을 넣었다. Spring Boot의 `SpringProfileIfNestedWithinSecondPhaseElementSanityChecker`가 이를 명시적으로 거부한다.

```
<springProfile> elements cannot be nested within an <appender>, <logger> or <root> element
```

핵심은 이게 **`addError`가 아니라 `addWarn`**이라는 점. 기동을 막는 쪽(`LogbackLoggingSystem.reportConfigurationErrorsIfNecessary()`)은 `Status.ERROR`만 모아 `IllegalStateException`을 던지므로, **WARN은 통과해서 앱은 그냥 뜬다.** 그래서 위반한 설정이 수년간 살아남을 수 있다 — 동작 보장은 없는 채로.

해결은 중첩을 뒤집는 것. 프로파일을 바깥에 두고 `<logger>`를 통째로 두 벌 쓴다.

```xml
<springProfile name="dev">
    <logger name="TCP_PACKET" level="INFO" additivity="false">
        <appender-ref ref="asyncPacketAppender"/>
        <appender-ref ref="packetConsoleAppender"/>
    </logger>
</springProfile>
<springProfile name="!dev">
    <logger name="TCP_PACKET" level="INFO" additivity="false">
        <appender-ref ref="asyncPacketAppender"/>
    </logger>
</springProfile>
```
