## 복사-붙여넣기로 만든 세터는 값의 출처가 어긋나도 타입만 같으면 컴파일·성공한다
태그: #java

가맹점 접수 수정에서 `representativePhone`(대표자 연락처)을 바꿔 보내면 API는 `"Success"`인데 DB 값이 안 바뀌었다. 원인은 도메인 갱신 메서드의 한 줄:

```java
this.representative      = vault.encrypt(command.representative());      // OK
this.representativePhone = vault.encrypt(command.representative());      // 버그: representativePhone() 이어야 함
```

윗줄을 복사해 아랫줄을 만들면서 `command.representativePhone()`으로 안 바꾼 것. 두 접근자 모두 반환 타입이 `String`이라 **컴파일러가 못 잡고, 더티체킹 커밋도 정상**이라 API는 성공한다. 다만 연락처 컬럼에는 매번 (변하지 않는) 대표자명 암호화값이 덮여, 사용자가 입력한 새 번호는 절대 저장되지 않는다.

같은 계열 필드(`xxx`/`xxxPhone`, `startAt`/`endAt`)를 복붙으로 세팅할 때 자주 나는 실수다. 파일 없는/있는 두 경로에 똑같이 복제돼 있었다 — **복붙 버그는 복붙된 곳마다 산다.** 이런 대칭 세터는 값 출처를 눈으로 대조하거나, 필드-커맨드 매핑을 MapStruct로 넘겨 손매핑을 없애는 게 안전하다.

## @RequestBody DTO에 없는 필드는 조용히 버려진다 — FE가 보내도 저장 안 되는 필드가 생긴다
태그: #spring

위 버그를 분석하다 발견. FE는 수정 요청에 34개 필드를 실어 보내는데, `/update`가 바인딩하는 `UpdateMerchantApplication` DTO는 19개뿐이었다. 나머지 15개(`settlementRatioBody`(수수료율), `businessType`, `picName`, `monthAmt` 등)는 Jackson 역직렬화가 **DTO에 없는 프로퍼티를 무시**(`FAIL_ON_UNKNOWN_PROPERTIES=false`)해 흔적 없이 사라진다. 에러도, 로그도 없다.

이번 요청은 그 15개가 대부분 빈값/`0`이라 티가 안 났지만, `settlementRatioBody`에는 실제 수수료율이 담겨 있었다. 즉 화면에서 수수료를 고쳐 저장하면 **성공 응답이 나와도 반영되지 않는다** — `representativePhone`과 똑같은 "성공인데 저장 안 됨"이 필드 누락으로도 발생한다.

교훈: "성공 응답 = 저장됨"이 아니다. request 스펙과 서버 DTO를 필드 단위로 대조해야 안다. 특히 등록 DTO에는 있는 필드가 수정 DTO에는 빠지면(여기선 수수료·업태·담당자 등) **등록은 되는데 수정은 안 되는** 비대칭이 조용히 생긴다.

## SslStream.AuthenticateAsClient(targetHost)의 targetHost가 인증서 이름 검증 기준이다 — TCP 접속 호스트와 별개다
태그: #dotnet #ssl

C# 소켓 테스터가 서버 SSL 인증서 교체 후 `Certificate error: RemoteCertificateNameMismatch`로 연결이 끊겼다. `RemoteCertificateNameMismatch`는 `SslPolicyErrors` 중 **"서버가 제시한 인증서의 CN/SAN이 검증 대상 호스트명과 불일치"** 만을 뜻한다(만료·체인 문제는 `RemoteCertificateChainErrors`로 따로 뜸).

핵심: `AuthenticateAsClient(targetHost)`의 `targetHost` 인자가 .NET이 인증서 SAN과 대조하는 이름이다. `client.Connect(host, port)`로 실제 접속하는 호스트와 **완전히 별개**다. 코드가 `AuthenticateAsClient("qr-road.com")`로 하드코딩돼 있었고, 구 인증서 SAN엔 `DNS:qr-road.com`이 있어 우연히 통과하던 것. 새 인증서 SAN에서 그 이름이 빠지자 바로 깨졌다.

```csharp
// 검증 콜백은 None이 아니면 무조건 실패 → 이름 불일치 시 핸드셰이크 중단
if (sslPolicyErrors == SslPolicyErrors.None) return true;
return false;
// 접속 대상(txtIpAddress)과 무관하게, 이 문자열이 SAN 검증 기준
((SslStream)sslStream).AuthenticateAsClient("qr-bank.co.kr"); // 새 인증서 SAN에 맞춤
```

함정: `ServicePointManager.ServerCertificateValidationCallback = delegate { return true; }`가 있어도 소용없다 — 이건 `HttpWebRequest` 전용이고, 직접 만든 `SslStream`은 자기 콜백만 탄다. 인증서 이름이 바뀌면 `AuthenticateAsClient`의 인자를 새 SAN에 맞춰야 한다.
