| 4 | 브라우저 없이도 `react-dom/server`의 `renderToStaticMarkup`로 PrimeReact 컴포넌트를 실제 렌더해 라벨/클래스 단위로 동작을 검증할 수 있음 (단, node_modules 해석을 위해 프로젝트 디렉터리에서 실행) |
| 5 | 1:N(단말기별) + 쓰기(CRUD) 데이터는 조회전용 카드에 끼우기보다 별도 탭으로 분리하는 게 데이터모델·UX·권한·확장성 측면에서 모범사례 |
| 6 | `/api/proxy`는 GET 요청 시 path 변수 치환 후 나머지 body를 `URLSearchParams`로 querystring 변환 → BE `@RequestParam` 전달 (page/size/processorId) |
| 7 | 화면에 보이는 컬럼이 없을 뿐 데이터(`code`)는 이미 응답·매퍼에 존재 → 컬럼 추가만으로 첫 조회부터 노출 (BE 변경 불필요) |

| 12 | PrimeReact `TabView`는 헤더가 기본 가로(좌우) 렌더 — POS-ID(terminal) 화면과 동일 패턴 |
| 13 | Mockito `any()` 는 내부적으로 `null` 을 placeholder 로 기록 → strict 모드 `PotentialStubbingProblem` 이 matcher 를 무시하고 raw 값(null) vs 실제 객체를 비교해 false positive 발생. `RestClient.RequestBodySpec.body()` 같은 Spring 내부 인터페이스 메서드에서 재현됨 → `lenient().when(...)` 으로 해당 stub 만 선택적으로 우회 |
| 14 | Mockito strict stubbing 3대 오류 패턴 — ① `UnfinishedStubbingException`: `when(A).thenReturn(mockB())` 처럼 `thenReturn` 인자 평가 중 내부에서 다시 `when()` 호출 시 발생 → 먼저 변수로 분리; ② `UnnecessaryStubbingException`: early return/throw 이후 접근 불가 필드 stub → 테스트별 최소 픽스처 분리; ③ `PotentialStubbingProblem`: 위 `body(any())` 패턴 |

| 16 | `useCallback` 의존성 배열에서 컴포넌트 state를 참조하는 경우 해당 state를 반드시 포함해야 클로저 stale 문제 없이 최신 값 사용 가능 |
