## 배운 것

| # | 내용 |
|---|------|
| 1 | 아웃바운드 HMAC 서명 시 `RestClient.body(Object)`에 객체를 직접 전달하면 내부에서 재직렬화되어 서명에 사용한 JSON과 달라질 수 있다. 반드시 `objectMapper.writeValueAsString(request)`로 사전 직렬화 후 `body(String)`으로 전달해야 byte 단위 일치를 보장한다 |
| 2 | 아웃바운드 메시지 구성 시 전체 URL이 아닌 path만 사용해야 한다. `URI.create(url).getPath()`로 추출 — 인바운드 `HmacAuthFilter`의 `request.getRequestURI()`가 path만 반환하는 것과 동일하게 맞춰야 서명 검증이 통과된다 |
| 3 | `HmacUtil.generateSignature()`는 `NoSuchAlgorithmException`, `InvalidKeyException` checked exception을 발생시키므로 호출 측에서 `try-catch`로 처리하고 `QRBankAppException`으로 래핑해야 한다 |
| 4 | 횡단 관심사(cross-cutting concern)는 비즈니스 로직과 분리하여 모든 경로가 수렴하는 단일 지점에 배치해야 한다. 파트너별 이벤트(`handleRefundRealTimeEvents`)와 전사 공통 이벤트(`IntegrationTransactionPublishEvent`)는 서로 다른 관심사이므로 섞으면 안 된다 |
| 5 | `RefundService`에서 전략 경로가 `return processRefundWithStrategy()`로 조기 반환하면 그 아래 공통 로직(횡단 관심사)이 실행되지 않는다. `if/else` 분기로 리팩토링하여 공통 코드를 마지막에 단일 실행하도록 구조를 바꿔야 한다 |
| 6 | 메서드명이 구현 세부사항을 드러낼 때(`notifyTerminalPaymentResult` = TCP Push 전용)는 책임이 확장될 때 이름도 함께 변경해야 한다. `handlePostPaymentEvents`처럼 추상적인 이름이 횡단 관심사 추가에 유연하다 |
| 7 | Facade + Delegation 패턴 적용 시 내부 전달 데이터가 여러 개일 때(`Payment` + `PaymentCallbackHolder`) private record(`ProcessResult`)로 묶어 반환하면 메서드 시그니처가 단순해진다 |
| 8 | `@Transactional` 메서드 내 `processType`은 try 블록 외부에서 미리 계산해야 catch 블록에서도 동일한 값을 사용할 수 있다. try 내부에서 계산하면 catch에서 접근 불가 |
