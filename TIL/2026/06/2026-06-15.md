## 배운 것

| # | 내용 |
|---|------|
| 1 | DDD Bounded Context + Hexagonal(Port-Adapter) 조합: BC를 최상위 패키지로 두고, 그 안에서 domain/port/in, out → application/service → infrastructure/adapter 구조를 적용하면 도메인 응집도와 의존성 역전을 동시에 확보할 수 있다 |
| 2 | `domain/port/out/`의 Outbound Port는 도메인 언어로만 표현해야 하며, 프레임워크 어노테이션(`@Service`, `@Component`) 및 인프라 타입(RestClient, JPA Entity) 참조 금지 |
| 3 | Application Service는 Port 인터페이스로만 주입받는다 — 구현체(Adapter) 클래스명이 Application 레이어에 등장하면 의존성 역전이 깨진 신호 |
| 4 | Adapter의 책임은 도메인 타입 ↔ 인프라 타입 변환(mapping)뿐 — 비즈니스 로직이 Adapter에 있으면 테스트 불가 영역이 생긴다 |
| 5 | `module-gateway`는 `module-api`에 의존 불가 → DTO에서 `AppBaseResponse`, `AppPaymentMethod` 등 module-api 타입 사용 금지, 자체 타입(String, 자체 record)으로 대체 |
| 6 | `cancelType(CANCEL/REFUND)` 결정 위치: RefundContext(공통 계약)가 아닌 연동사 서비스 내부 — OqmsRefundService 패턴 참조 |
