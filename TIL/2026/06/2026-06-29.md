| 15 | **DIP 핵심 = "호출 방향"과 "소스 의존 방향"의 분리**. 런타임 호출은 Service→인프라 그대로지만, 인터페이스(port)를 핵심이 소유하면 소스 의존이 인프라→핵심으로 역전됨. 콘센트 비유: 쓰는 쪽(노트북)이 규격을 소유, 만드는 쪽(충전기)이 맞춘다 |
| 16 | **헥사고날 위반은 "repo가 많아서"가 아니라 "app이 JPA 엔티티/repo를 직접 import"하는 것**. 판단 기준은 import 목록 — `com.road.qr.internal.database.*`가 application 계층에서 사라지는 게 적용 증거 |
| 17 | **"전용 adapter 하나에 다 넣기"는 God Port 안티패턴**. port는 use-case 욕구 기준으로 응집되게 분리(ISP), adapter 내부에서 여러 repo 조합은 OK. "port 분리"와 "adapter가 repo 묶기"는 다른 차원 |
| 18 | **use-case형 port(B방식)가 효율적인 이유**: 데이터 수집 잡일을 응집된 port 뒤로 밀어 ① Service는 오케스트레이션만 ② DTO 경계로 엔티티 누수 차단 ③ adapter가 단일 join 등 최적화 자유 ④ mock 수 급감으로 테스트 단순화 |
| 19 | adapter는 `@Transactional` 없이 호출자(Service) 트랜잭션에 참여 → 저장→AFTER_COMMIT 이벤트 발행 흐름 보존. 경계 이동 시 트랜잭션 전파 확인 필수 |
