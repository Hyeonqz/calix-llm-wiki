## 배운 것

| # | 내용 |
|---|------|
| 1 | Spring 6.1 `RestClient.RequestBodySpec.body(Object)` 는 **default 메서드** → `when(mock.body(any()))` 사용 시 Mockito 가 메서드를 직접 호출해 내부 default 구현이 실행됨. 그 결과 argument matcher 가 올바르게 등록되지 않아 실제 호출 `body(Secta9Inputs(...))` 와 매칭 실패 → null 반환 → NPE |
| 2 | Mockito 에서 default 인터페이스 메서드를 stub 할 때는 `doReturn(val).when(mock).method(matcher)` 를 사용해야 한다. `doReturn` 방식은 메서드를 호출하지 않고 직접 스텁을 등록하므로 default 메서드 문제를 완전히 우회 |
| 3 | `lenient().when()` 은 `PotentialStubbingProblem` 경고를 억제할 뿐, 실제 stub 매칭 자체를 고치지는 않는다. 경고가 사라져도 stub 이 null 을 반환할 수 있음 |
| 4 | 고정 포맷 복합 문자열(`paymentCompanyId` 38자)을 substring 으로 파싱할 때, DB 컬럼 길이 제약이 없으면 불량 데이터가 들어올 수 있다. 파싱 전 항상 길이 검증이 필요하며, 배치에서는 단건 오류가 전체를 중단시키지 않도록 스킵 처리가 필수임 |
| 5 | `Mockito.lenient()` 는 `LenientStubber` 반환 → `given()` 없음. BDD 스타일 lenient 는 `BDDMockito.lenient()` 사용. 또는 클래스 레벨 `@MockitoSettings(strictness = Strictness.LENIENT)` 이 가장 깔끔한 해결책 |
| 6 | `given(repo.findById(1L)).willReturn(mockHelper())` 처럼 `willReturn()` 인자 안에서 또 다른 `given()` 을 시작하면 `UnfinishedStubbingException` 발생 (Mockito 힌트 #3). mock 을 변수로 먼저 완성한 뒤 `willReturn(variable)` 에 넘겨야 함 |
| 7 | **3-tier 수수료 구조**: `pymt_cmpy_rat`(큐뱅 원가) → `mer_pymt_cmpy_rat`(가맹점 수수료) → `agncy_pymt_cmpy_rat`(대리점 원가). 배치는 NOT EXISTS 패턴으로 `cre_tm` 기준 최신 레코드 1건만 선택함 |
| 8 | **NOT EXISTS + cre_tm 우선 원칙**: 수수료 적용 시작일(`cmms_aply_sta_tm`)이 더 이전이어도 `cre_tm`이 최신이면 그 레코드가 적용된다. 즉 2026-06-25에 등록한 sta_tm=2026-02-01 레코드가 2026-03-01에 등록한 sta_tm=2026-03-01 레코드를 이긴다 |
| 9 | **CROSS JOIN + UNION ALL 실전 패턴**: `FROM merchant m CROSS JOIN (SELECT 'UnionPay' AS pymt_cmpy ... UNION ALL ...) v WHERE m.agency_id = 2` — N 가맹점 × M 결제수단을 루프 없이 한 번의 SQL로 INSERT 처리 |
| 10 | **UNION vs UNION ALL**: UNION은 내부 DISTINCT + 정렬 발생으로 느림. 중복이 없다고 확신할 수 있으면 항상 UNION ALL 사용. 인라인 고정값 테이블 생성 시 UNION ALL이 표준 패턴 |
| 11 | **`SELECT ROW_COUNT()`**: INSERT 직후 동일 실행 컨텍스트에서만 유효. DB 툴이 문장을 개별 실행하면 항상 0 반환. 해결책: 전체 블록을 한 번에 드래그 선택 후 실행, 또는 INSERT 전후 COUNT(*) 비교 |
| 12 | **INSERT 방식 vs UPDATE in-place**: 수수료 변경 시 기존 행을 UPDATE하지 않고 신규 INSERT. NOT EXISTS가 `cre_tm` 최신 행을 선택하므로 새 행이 자동으로 우선 적용. 히스토리 추적도 자연스럽게 됨 |

## 이슈 / 트러블슈팅

| # | 이슈 | 해결 |
|---|------|------|
| 1 | `when(requestBodySpec.body(any())).thenReturn(requestBodySpec)` — `PotentialStubbingProblem` 발생, `lenient()` 로 경고는 사라졌으나 여전히 `body()` 가 null 반환 → NPE | Spring 6.1 `body(Object)` 가 default 메서드임을 확인. `doReturn(requestBodySpec).when(requestBodySpec).body(any(Object.class))` 로 교체하여 해결 |
| 2 | EZL 배치 `IndexOutOfBoundsException: start 0, end 12, length 4` — `paymentCompanyId` 가 4자로 저장된 불량 데이터 레코드로 인해 배치 전체 중단 | substring 호출 전 `paymentCompanyId.length() != 38` 검증 추가, 불량 건은 warn 로그 후 `continue` 로 스킵 |
| 3 | `lenient().given()` — `cannot resolve method 'given' in 'LenientStubber'` 컴파일 오류 | `Mockito.lenient()` 대신 클래스 레벨 `@MockitoSettings(strictness = Strictness.LENIENT)` 적용 |
| 4 | `given(repo.findById(1L)).willReturn(mockHelper())` — `UnfinishedStubbingException` 런타임 오류 (`willReturn` 인자 안에서 중첩 `given()` 호출) | mock 을 변수로 먼저 생성 후 `willReturn(variable)` 에 전달하는 방식으로 수정 |
| 5 | 실제 DB column명이 인수인계 문서와 달랐음 — `line_pay_cmms_code` (X) → 실제는 `linepay_cmms_type_code` | Entity 파일(`Merchant.java`) 직접 확인 후 SQL 수정. PaymentCompanyCode DB 저장값도 `WeChat`(X) → `Wechat`, `LINE Pay`(X) → `LINEPay` 로 확인 |
| 6 | `SELECT ROW_COUNT()` 가 0 반환 — INSERT 후 별도 실행 시 컨텍스트가 달라짐 | INSERT 전후 `SELECT COUNT(*) FROM table WHERE agency_id = 2` 비교 방식으로 검증. 또는 전체 블록 선택 후 한 번에 실행 |
