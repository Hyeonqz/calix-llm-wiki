## 배운 것

| # | 내용 |
|---|------|
| 1 | "시간당 평균"은 TPS가 아니다. TPS = 초 단위 동시 처리량이라 1시간 평균으로 내면 피크가 평탄화되어 과소평가됨 |
| 2 | 피크 확정은 단일 MAX가 아니라 분포의 **p99**로 잡아야 함 (MAX는 outlier, p99가 상시 견뎌야 할 현실 피크) |
| 3 | 피크 측정 3단계: 시간대 좁히기 → 초 단위 실측 → 초당 처리량 분포로 p99 확정 |
| 4 | 노티 지연을 `Thread.sleep`으로 구현하면 컨슈머 스레드(`concurrency=2`)를 묶어 처리량 = `concurrency/지연시간`으로 급락, Kafka 리밸런싱 폭주 위험. deferred 발송(Redis ZSet/DB 예약)은 지연시간이 처리량에 무관 |
| 5 | "3~5초 지연"은 근본 해결이 아니라 **확률적 완화책**. 동기 결과가 지연 시간을 초과하면 중복 재발 → 근본 해결은 수신 측 멱등(idempotency) 처리 |
| 6 | 협상은 부하 명분(측정 후 무력화됨)이 아니라 "설계 적절성 + 책임 분담"으로 프레이밍하는 게 효과적. 양측이 한 겹씩 방어(노티 억제 + 수신측 dedup)가 가장 견고 |
| 7 | SpringBoot 3→4는 메이저 변경: Jackson 2→3(`com.fasterxml.jackson`→`tools.jackson`), JUnit 5→6, Hibernate 7 동반 → "코드 변경 0"은 불가능, 목표는 "동작 변경 0 + 테스트로 증명"으로 재정의해야 함 |
| 8 | 레거시 업그레이드 모범사례: Characterization/Golden Master 테스트로 현재 동작 박제 → 단, 기존 버그까지 박제되므로 KNOWN-BUG 명시 분리. 커버리지 %는 허영지표 → PITest 뮤테이션으로 보완 |
| 9 | QueryDSL 5.0.0은 유지보수 중단 → SpringBoot4에선 openfeign 포크(querydsl-7.0) 필요. 1차 깨짐은 런타임이 아니라 APT(Q클래스 생성) 단계 |
| 10 | Jacoco는 콘솔에 커버리지를 안 찍고 파일(HTML/XML)만 생성. `build -x test`로 빌드하면 test 미실행→exec 없음→커버리지 0. 커버리지는 test가 실제 실행돼야 나옴 |
| 11 | 멀티모듈 Jacoco: 모듈별 `jacocoTestReport`(자동)와 root 합산 `JacocoReport` 태스크는 공존 가능. 합산은 build 그래프에 안 묶이므로 명시 호출(`jacocoAggregateReport`) 필요 |
| 12 | `/etc/hosts`는 호스트명→IP 매핑만 가능, **IP→IP 리다이렉트는 불가**. 순수 IP 서비스(192.168.20.x)는 WSL 루프백 IP 별칭(`ip addr add 192.168.20.250/32 dev lo`)으로 우회해야 함 |
| 13 | dev 프로파일은 DB 비밀번호를 vault가 아닌 application-dev yml에서 직접 읽음(`DataSourceConfig`가 dev면 `getDevDataSource()` 분기) → dev용 vault DB 시드 불필요 |
| 14 | Spring OAuth2 resource server `issuer-uri`는 부팅 시 `.well-known/openid-configuration`을 eager fetch → realm 없으면 부팅 실패. 4개 모듈 중 scheduler만 keycloak.yml import하므로 부팅 필수 realm은 `qrgw-scheduler` 하나 |
| 15 | `VaultService.getSecretData()`는 `read(path).getData().get("data")` 구조 = KV v2 시맨틱. dev 모드 Vault는 `secret/`만 자동 마운트 → `mpos/`·`dqms/` 엔진 별도 마운트 필요 |
| 16 | Firebase(`GoogleCredentials.fromStream`)·APNs(`ApnsSigningKey.loadFromInputStream`)는 부팅 시 키를 **파싱만** 하고 네트워크 검증 안 함 → openssl로 만든 구조상 유효한 더미 키(RSA service_account JSON / EC P-256 .p8)로 부팅 통과 가능 |
| 17 | 앱 yml 무수정 원칙: docker-compose 포트를 dev 값(vault 443/keycloak 38080/kafka 10001~)에 정렬하면 앱 설정 무수정으로 폐쇄망과 100% 동일 유지 |
| 18 | 다운로드 "진짜 진행률"은 FE만으론 불가 — 서버가 중간 상태를 노출(Redis polling/SSE)해야 함. 단일 쿼리 통째 반환 구조에선 단계별(staged) 진행률(10/30/80/100)로 "100%=실제 완료"만 보장 |
| 19 | SSE는 표준 EventSource가 커스텀 헤더 불가 → Keycloak Bearer 인증 프록시 환경에선 FE 구현 비용 큼. Polling은 기존 ApiService 그대로라 토큰 자동 해결 → 비동기 다운로드엔 Polling이 실용적 |
| 20 | 화면별 다운로드 방식 상이: 6개는 서버 download API 보유(폴링 가능), 해외/유니온페이는 로드된 `products`를 클라이언트서 즉시 export(서버 API 없음→폴링 대상 아님). 기능 조사로 발견해 enum·스코프 보정 |
| 21 | DB 엔티티 수정 금지 제약에서도 엔티티에 이미 컬럼(`english_name/address/zip_code`)+`@Setter`가 있으면 어댑터 setter로 갱신 → 엔티티/스키마 변경 0으로 수정 기능 구현 |
| 22 | 수정자 이력은 DB 컬럼이 없을 때 수수료 등록 패턴처럼 FE가 `session.user.name`을 `userNm`으로 전송 → BE `log.info` 로깅만으로 추적 |
| 23 | writing-plans는 placeholder 없이 전체 코드를 담아야 폐쇄망에서 문서만으로 실제 로직 100% 재현 가능(build/test만 내부 Nexus) |
