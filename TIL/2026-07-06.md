## 배운 것

| # | 내용 |
|---|------|
| 1 | QueryDSL `fetchOne()`은 결과가 2건 이상이면 `NonUniqueResultException` — 존재 확인만 필요하면 `fetchFirst()`(내부적으로 limit 1)를 써야 안전 |
| 2 | 이력 존재 체크는 엔티티를 로딩할 필요 없이 `selectOne().from().where().fetchFirst()` 결과의 null 여부로 처리 → 불필요한 hydration 회피 |
| 3 | "하루 1회 실행" 멱등 제어는 batchType만으로 부족 — 동일 batchType이 날짜별로 누적되므로 `targetDate`까지 조건에 넣어야 날짜 단위로 스코프됨 (그래야 다건 누적 자체가 예방되고, 그럼에도 `fetchFirst`로 이중 안전장치) |
| 4 | `status='DONE'`만 필터하면 이전 ERROR 이력은 재실행을 막지 않음 → 실패 건은 같은 날짜라도 재시도 가능 (의도된 설계) |
| 5 | 메서드명(`existsBy~`)과 반환 의미(`== null`, 즉 "없을 때 true")가 반대면 호출부 변수명(`alreadyExecuted`)까지 역으로 오염됨 — 이름-의미 정합이 유지보수/폐쇄망 재구현 시 오독을 예방 |
| 6 | `@Transactional` 없는 메서드에서 `JpaRepository.save()`는 자체 트랜잭션으로 커밋 후 **detached 엔티티**를 반환 → 이후 setter 변경은 영속성 컨텍스트 밖이라 dirty checking flush가 안 됨. 명시적 재-save 또는 메서드에 `@Transactional` 필요 |
| 7 | 동일한 "세팅→성공처리" 코드라도 트랜잭션 경계 유무로 영속 결과가 갈림 — REQ(`upload`, `@Transactional`)는 저장되고 RES(`process`, 미부여)는 유실. 스냅샷 코드 리뷰 시 트랜잭션 애노테이션 유무를 반드시 대조 |
| 8 | 실패 이력 보존 관점에서 명시적 `updateExecutionHistory()`가 `@Transactional`보다 안전 — `@Transactional`은 후속 예외 시 INIT 이력 행까지 롤백되지만, 명시적 save는 성공분만 반영하고 실패 시 INIT 행이 남음 |
