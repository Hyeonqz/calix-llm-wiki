## 배운 것
- **ADR 문서는 코드베이스와 대조 검증이 필수**
  - 문서에 적힌 As-Is 구조가 실제 코드와 일치하는지 Grep/디렉토리 탐색으로 확인해야 함
  - 이번에 발견한 누락: `module-api/shared/`의 `HmacAuthFilter`, `ApiGlobalExceptionHandler`가 `gateway.app`을 import하고 있었는데 영향 범위에서 빠져 있었음
- **배포 모듈 내 비즈니스 로직 배치 기준**
  - "배포 모듈에 비즈니스 로직을 두면 안 된다"는 원칙의 정확한 의미: **공유되는** 로직만 라이브러리 모듈로 분리
  - 단독 사용 + 공유 불필요 + 내부 레이어 구조 유지 시 배포 모듈에 두는 것이 타당
- **DDD 레이어드에서 컨텍스트 간 의존 규칙**
  - 같은 모듈 내 Bounded Context(`app/`, `pos/`, `web/`) 간 직접 참조 금지
  - 횡단 관심사(`shared/`)만 모든 컨텍스트 참조 가능
  - 컨텍스트 간 통신은 Kafka 이벤트 또는 공유 라이브러리 모듈 경유
- **서블릿 필터에서 예외를 throw하면 `@ControllerAdvice`가 잡지 못한다**
  - `@ControllerAdvice`는 `DispatcherServlet` 내부에서 발생한 예외만 처리
  - 서블릿 필터는 `DispatcherServlet` 밖에서 동작하므로, throw한 예외는 Tomcat이 직접 처리
  - 따라서 필터에서는 `response.getWriter().write()`로 직접 응답을 작성하고 `return`하는 방식이 올바름
- **FilterRegistrationBean URL 패턴과 shouldNotFilter()의 관계**
  - `FilterRegistrationBean`의 URL 패턴은 서블릿 컨테이너 레벨에서 필터 호출 여부를 결정
  - `OncePerRequestFilter.shouldNotFilter()`는 필터 내부에서 추가 필터링 담당
  - URL 패턴을 좁게 잡으면 `shouldNotFilter()` 로직이 무의미해질 수 있음 (이번 버그의 원인)
- **필터에서 Path Variable 추출 방법**
  - 필터 단계에서는 Spring MVC의 `@PathVariable` 바인딩이 아직 수행되지 않음
  - URI를 직접 파싱하는 방식이 가장 간단: `request.getRequestURI().split("/")`
  - RequestBody에서 추출도 가능하지만 (`CachedBodyHttpServletRequest` + `ObjectMapper`), JSON 파싱 실패/필드 누락 등 방어 코드가 필요해서 URI 파싱이 더 적합

## 이슈 / 트러블슈팅
- **HmacAuthFilter가 실제로 동작하지 않던 문제**
  - 원인: `FilterRegistrationBean` URL 패턴이 `/endpoints/app/linepay/*`로 설정되어 있었고, `shouldNotFilter()`가 linepay를 제외하는 로직이라 어떤 요청에도 HMAC 검증이 실행되지 않았음
  - 해결: URL 패턴을 `/endpoints/app/*`로 변경, 세부 경로 필터링은 `shouldNotFilter()`에 위임
- **RequestBody에서 transactionNo 추출 시 NPE**
  - `objectMapper.readTree(body).get("transactionNo")`가 null 반환 → `.asText()` 호출 시 NPE
  - body가 비어있거나 해당 필드가 없는 경우 null-safe 처리 필요
