## 배운 것

| # | 내용 |
|---|------|
| 1 | `Mockito.lenient()`는 `LenientStubber`를 반환하며, `given()`이 없고 `when()`만 있다 → `lenient().when().thenReturn()` 형식 사용 |
| 2 | BDDMockito 스타일(`given/willReturn`)과 Mockito 스타일(`when/thenReturn`)은 혼용 불가 — `lenient()` 체이닝에서는 `when` 계열만 사용 가능 |
| 3 | Controller ↔ Service 간 path variable 전달: `@PathVariable` 값은 Command 객체에 별도 필드로 담아 Service 레이어에서 비즈니스 검증 수행 (Controller에서 검증하지 않음) |
| 4 | Claude Code `settings.local.json`의 `defaultMode: "bypassPermissions"`를 설정하면 모든 툴 실행 시 승인 프롬프트 생략 가능 — 위험한 명령은 `deny` 배열로 명시적 차단 |
| 5 | Claude Code 서브에이전트(Agent 도구)로 오케스트레이터→BE/FE 병렬 작업 분배 가능 — 별도 터미널/세션 불필요, 컨텍스트 격리 |
| 6 | 공유 컴포넌트 변경은 모든 소비 화면(여기선 가맹점 드롭다운=5개 화면)에 일괄 반영되므로 영향 범위를 먼저 확인해야 함 |

## 이슈 / 트러블슈팅

| # | 이슈 | 해결 |
|---|------|------|
| 1 | `lenient().given(merchant.getId()).willReturn(10L)` → `Cannot resolve method 'given' in 'LenientStubber'` 컴파일 에러 | `lenient().when(merchant.getId()).thenReturn(10L)` 으로 변경 — `LenientStubber`는 `when()`만 제공 |
| 2 | `AppPaymentCommand.create()` 호출부가 4개 파라미터 기준으로 작성되어 있었는데 5번째 `pathTransactionNo` 추가 후 전체 컴파일 오류 | 테스트 내 모든 `create()` 호출부에 적절한 5번째 인자 추가 (DPM happy path → transactionNo 동일값, SPM/UNKNOWN → null) |
| 3 | 보조 FE 에이전트가 계획 범위 밖 코드 613줄(LinePay MID 상세 기능) 생성 | 커밋하지 않고 `git stash`로 격리 보관(검토용), 계획상 5개 파일만 반영 |
