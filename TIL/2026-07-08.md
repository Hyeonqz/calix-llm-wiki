# TIL - 2026-07-08 (수)

## QRBank 에러코드는 {도메인2}{채널1}{유형2}{시퀀스3} 체계 — 입력값 검증 실패는 WC(Wrong Condition)로 정의
태그: #spring-boot

QR 결제 부가세 검증(`vat ≤ totalAmount × 10%`) 실패 시 `IllegalArgumentException`을 던지던 것을 `QRBankException` + 정의된 에러코드로 교체했다. 에러코드 체계를 분석해 보니 `{도메인 2자리}{채널 1자리}{유형 2자리}{시퀀스 3자리}` 구조였다 (예: `CMCWC011` = CM 공통 + C Common 채널 + WC Wrong Condition + 011).

핵심 판단 기준:
- 클라이언트 입력값이 조건에 안 맞는 경우 → `ER`(서버 오류)이 아닌 `WC` + 심각도 `FAIL` (서버 장애가 아닌 요청 거절)
- 기존에 `IllegalArgumentException`을 쓰고 있었다는 것 자체가 "잘못된 입력" 성격이라는 힌트
- 신규 코드 정의 전 가장 가까운 선례를 찾는다 — 금액 간 관계 검증인 `REFUND_AMOUNT_BIGGER_THAN_PAYMENT_AMOUNT("CMCWC004")`가 같은 시리즈에 있어 `CMCWC011`로 배정
- enum 이름은 실패 이유가 드러나게 (`VAT_EXCEEDS_LIMIT` > `INVALID_VAT_AMOUNT`)

```java
// SystemConstants.QRBankErrorCode
VAT_EXCEEDS_LIMIT("CMCWC011", "부가세는 결제금액의 10%를 초과할 수 없습니다.", FAIL),

// PaymentDpmServiceImpl — before: throw new IllegalArgumentException("vat cannot exceed 10%...")
if (!isValidVat(intermediateInput.getTotalAmount(), intermediateInput.getVat()))
    throw new QRBankException(SystemConstants.QRBankErrorCode.VAT_EXCEEDS_LIMIT);
```

## 공용 라이브러리 enum 변경은 원본 리포·스냅샷 동기화 + 폐쇄망 선배포 순서가 필수
태그: #devops

`com.road.qr.internal.constants.SystemConstants`는 서비스 리포가 아니라 별도 리포(core-lib, 폐쇄망에선 dqms-persistence-lib jar)가 원본이다. 서비스 리포의 `apps/core/`는 참조용 소스 스냅샷일 뿐 빌드 대상이 아니다 (settings.gradle에 없음).

따라서 라이브러리 상수 하나를 추가해도 수정 지점은 3곳:
1. core-lib 원본 (실제 컴파일 대상 — `./gradlew compileJava --offline`로 검증)
2. 서비스 리포의 스냅샷 사본 (참조 일관성 유지)
3. 서비스 코드의 사용처

폐쇄망 이관 시 순서 중요: 라이브러리를 먼저 수정·배포(1.0.235 → 이후 버전)한 뒤에 서비스 변경을 적용해야 컴파일이 깨지지 않는다. 서비스 모듈 전체 빌드가 스냅샷 export 누락으로 막혀 있어도, 사용처 변경이 기존 코드와 동일 패턴(같은 생성자·enum 타입 사용례가 같은 파일에 존재)이면 타입 안전성은 근거를 갖고 판단할 수 있다.
