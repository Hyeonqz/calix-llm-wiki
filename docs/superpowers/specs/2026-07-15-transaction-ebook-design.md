# 트랜잭션 전자책 『약속된 데이터』 — 설계 문서

날짜: 2026-07-15
상태: 승인 대기

## 목적

데이터베이스 **트랜잭션**을 역사부터 현재까지, 근본 원리와 실무 사례로 풀어내는
**그림이 있는 전자책**을 만든다. 만화(컷·말풍선)가 아니라, AI 삽화가 곁들여진 해설서다.

세 가지 목표를 동시에 만족한다:
- **블로그 게시** — 이 위키 사이트 안에서 웹으로 읽는다.
- **외부 배포** — 링크 하나로 누구나 공개 접근.
- **이력서 기재** — "트랜잭션 전자책 기획·집필·배포" + PDF 출간물.

## 목표 독자

RDB/DB에 본격 입문하기 전, "왜 트랜잭션이 필요한가"를 근본부터 이해하고 싶은
예비/주니어 개발자. 비전공자도 초반부는 읽을 수 있는 눈높이.

## 설계 결정 (확정)

1. **제목**: 『약속된 데이터 — 트랜잭션은 왜 태어났는가』
2. **라우트**: `/book` (Nextra `content/book/`가 `app/[[...slug]]`로 자동 렌더 —
   신규 라우트 코드 0줄)
3. **공개 범위**: **외부 공개** (인증 게이트 없음). `/til`·`/study`와 달리 막지 않는다.
4. **콘텐츠 저장**: **`content/book/`** 폴더에 장별 마크다운 1파일 = 단일 소스.
   (Nextra 4는 `content/` 트리만 렌더하므로 반드시 `content/` 하위여야 한다.
   루트 `book/`는 `importPage`가 잡지 못한다 — 검증 완료.)
5. **PDF**: 사용자가 직접 제작해 첨부한다. **자동 생성 파이프라인 불필요.**
   전권을 한 페이지에 이어붙인 **`/book/full` 통합 뷰**를 두어, 사용자가 브라우저
   인쇄(print-to-PDF)로 전권 PDF를 뽑을 수 있게 한다. 뽑은 PDF는
   `public/book/promised-data.pdf`에 넣고, `content/book/index.md` 상단 다운로드
   버튼(`/book/promised-data.pdf`)에서 링크한다.
6. **삽화**: 각 장 상단에 그 장을 요약하는 **웹툰식 요약 이미지 1컷** + 본문 중간 개념 삽화.
   프롬프트는 Claude가 작성 → 사용자가 AI 툴로 생성 → **`public/images/book/{장번호}/`**에
   넣고 `/images/book/{장번호}/...`로 연결. (정적 자산은 `public/`에서만 서빙된다 —
   기존 블로그가 `/images/blog/...`를 쓰는 것과 동일 규칙.)
7. **코드**: SQL 예제 + Spring Boot/Java 예제 둘 다 포함.
8. **집필 방식**: 18장을 한 번에 쓰지 않고 **장 단위 반복**(초안 → 리뷰 → 삽화 프롬프트).

## 페이지 포맷 (각 장)

각 장은 다음 구조를 따른다:

```markdown
---
title: 1장. 장부 앞의 회계사
---

![1장 요약: 촛불 아래 두 장부에 동시에 기록하는 회계사](/images/book/01/summary.png)   ← 장 상단 웹툰식 요약 이미지 (필수, alt 텍스트로 내용 서술)

# 1장. 장부 앞의 회계사 — 트랜잭션의 선사시대

## 배경
(왜 이 이야기가 필요한가 — 도입 훅)

## 스토리
(역사적 일화·실무 사례·비유로 몰입)

## 핵심
(개념 설명 + 개념 삽화 + SQL/Java 예제 박스)

## 정리 & 생각해볼 질문
```

- **장 상단 요약 이미지**: 그 장 전체 내용을 한 컷으로 압축한 웹툰식 그림. 독자가
  장을 열자마자 "이 장은 무슨 이야기인지" 시각적으로 파악.
- **본문 흐름**: `배경 → 스토리 → 핵심` 3단 고정.
- **코드 박스**: 핵심 절에 실행 가능한 SQL, 필요 시 Spring Boot/Java 스니펫.

## 목차 (6부 18장, 약 260~300쪽)

여는 글 — 당신의 잔고가 사라진 밤

**제1부 — 트랜잭션이 없던 세계 (역사·문제의식)**
- 1장. 장부 앞의 회계사 — 트랜잭션의 선사시대
- 2장. 파일과 씨름하던 프로그래머들 — 초기 DBMS의 등장
- 3장. System R과 하나의 단어 — "Transaction"의 탄생
- 4장. SQL 표준이 세운 약속 — 언어가 된 트랜잭션

**제2부 — ACID, 네 개의 약속 (근본 원리)**
- 5장. A — 전부 아니면 전무 (원자성)
- 6장. C — 규칙은 깨지지 않는다 (일관성)
- 7장. I — 서로를 방해하지 않는 척 (격리성)
- 8장. D — 한번 약속하면 영원히 (지속성)

**제3부 — 동시성이라는 전쟁터 (격리 수준·이상 현상)**
- 9장. 유령과 오염 — 동시성이 부르는 괴현상들
- 10장. 네 개의 문 — 격리 수준의 사다리
- 11장. 표준이 말하지 않은 것 — 스냅샷 격리와 현실의 DB

**제4부 — 질서를 지키는 장치들 (동시성 제어·복구)**
- 12장. 자물쇠를 채우다 — 락(Lock)과 비관적 제어
- 13장. 시간을 여러 겹으로 — MVCC의 우아함
- 14장. 믿음의 문제 — 낙관적 제어와 충돌 감지
- 15장. 재난 후의 복구 — WAL과 로그의 힘

**제5부 — 현장의 트랜잭션 (실무·장애)**
- 16장. 서로를 붙든 채 멈추다 — 데드락과 롱 트랜잭션
- 17장. 프레임워크가 숨긴 트랜잭션 경계 — 커넥션 풀 함정과 `@Transactional`의 진실
  (커넥션 풀 고갈 실전 장애 + Spring `@Transactional` 전파·롤백 규칙·self-invocation 프록시 함정)

**제6부 — 트랜잭션의 확장과 미래**
- 18장. 경계를 넘는 약속 — 분산 트랜잭션, 그리고 왜 근본인가

닫는 글 — 다시, 커밋 앞에서

서사 곡선: **역사(왜) → 원리(무엇) → 구현(어떻게) → 현장(정말?) → 확장(그다음)**.
1장의 "사라진 잔고"로 회귀하는 수미상관.

## 간판 실무 사례 3개 (책 전반에서 재활용)

송금 대신, 각기 다른 트랜잭션 개념을 대표하는 3개 사례를 관통 예시로 쓴다.

1. **콘서트/항공 좌석 예약 — "마지막 한 자리"**
   - 대표 개념: 격리성·락·Lost Update / 오버부킹(oversell)
   - 두 명이 동시에 마지막 1석 결제 → 둘 다 성공하면 오버부킹. 비관적 락 vs
     낙관적 락(`@Version`) 비교의 교과서. (주로 7·10·12·14장)

2. **이커머스 주문 체크아웃 — "재고 + 결제 + 쿠폰"**
   - 대표 개념: 원자성·부분 실패·트랜잭션 경계
   - 재고 차감 + 결제 승인 + 쿠폰 소진이 원자적이어야. 외부 PG 호출이 끼면서
     17장(커넥션 풀 함정)·18장(분산/Saga)으로 연결. (주로 5·17·18장)

3. **크래시 직후의 잔고 — "서버가 죽어도 데이터는 산다"**
   - 대표 개념: 지속성(D)·WAL·크래시 복구
   - "결제 완료"를 본 직후 서버 전원이 나갔다. 재부팅 후에도 그 데이터는 살아있어야 한다.
     WAL(로그 먼저) → redo/undo → 체크포인트가 어떻게 이걸 보장하는지. (주로 8·15장)

> 3개 사례가 **① 격리성/락(좌석) · ② 원자성/부분 실패(체크아웃) · ③ 지속성/복구(크래시)**로
> 서로 다른 축을 대표하도록 구성했다. Lost Update(순진한 Read-Modify-Write) 자체는
> 사례 ①(좌석)의 오버부킹 안에서 함께 다룬다.

## 렌더링 (웹 `/book`)

- `content/book/`을 `app/[[...slug]]`가 자동 렌더 — **신규 라우트 코드 0줄.**
  `/book`(← `content/book/index.md`), `/book/01-ledger-accountant` 등 전 라우트가 자동 생성.
- 사이드바(`_meta.js`)에 부/장 순서 노출. prev/next 내비게이션은 nextra-theme-docs가 자동 제공.
- `index.md` 상단에 **PDF 다운로드 버튼**(`/book/promised-data.pdf` 링크).
- 삽화는 인라인 반응형 이미지(`![alt](/images/book/...)`), 모바일에서 `max-width:100%`.
  모든 삽화에 내용을 서술하는 alt 텍스트 필수(접근성).
- **인증 게이트 없음** — `/til`·`/study`처럼 `proxy.js` matcher에 추가하지 **않는다**.
- **`/book/full` 통합 뷰**: 유일하게 커스텀 코드가 필요한 부분. `content/book/*.md`를
  순서대로 읽어 한 페이지로 이어붙이는 소규모 라우트(`app/book/full/page.jsx` 등).
  print CSS로 인쇄 시 사이드바·내비를 숨긴다. 구현 방식은 계획에서 확정.

## 이력서 보강 (선택 companion, stretch)

책이 코어다. 아래는 엔지니어링 신호를 더하고 싶을 때의 **완성 후 별도 트랙**이며,
이번 스펙의 필수 범위가 아니다(과한 선행 투자 방지):

- 간판 사례를 **실행·검증 가능한 데모 저장소**로 만든다. 실제 DB로 race condition을
  재현하는 실패 테스트 → 낙관적 락(`@Version`)/원자적 갱신으로 수정 → 통과 테스트.
- 효과: "전자책을 썼다"가 "동시성 문제를 재현·수정해 테스트로 증명했다"로 강화된다.
- 책 본문 코드 박스에서 이 저장소를 링크해 상호 보강.

## 삽화 워크플로우

1. **스타일 가이드 선고정**: `docs/book/style-guide.md`에 화풍·색감·캐릭터(회계사,
   개발자 등 반복 등장 인물)를 정의. 모든 삽화 프롬프트가 이걸 참조해 일관성 유지.
2. **장별 요약 이미지**: 각 장 초안 완성 후 Claude가 "이 장 요약 웹툰 1컷" 프롬프트 작성.
3. **개념 삽화**: 핵심 절에서 시각화가 필요한 개념마다 프롬프트 작성.
4. 사용자가 AI 툴로 생성 → `public/images/book/{장번호}/`에 저장 → `/images/book/{장번호}/...`로 연결.
5. **1장에서 일관성 실증(de-risk)**: 18장 전량에 커밋하기 전, 1장의 요약+개념 3~4컷으로
   캐릭터·화풍 일관성이 실제로 유지되는지 먼저 검증. 안 되면 반복 캐릭터를 포기하고
   추상 개념 삽화 위주로 스코프를 줄이는 fallback을 택한다.

## 집필 워크플로우 (운영 방식)

- 장 단위로 진행: 초안 작성 → 사용자 리뷰 → 수정 → 삽화 프롬프트 → 다음 장.
- 한 번에 전권을 쓰지 않는다(품질·일관성·리뷰 가능성 확보).
- 순서는 목차 순(1장부터)을 기본으로 하되, 사용자가 원하는 장부터도 가능.

## 테스트 / 검증

- **스모크 테스트 먼저**: `content/book/index.md` 스텁 1개 + `content/_meta.js`에 `book`
  추가 후 `npm run build`로 `/book` 렌더 확인(C1 회귀 방지). 통과해야 본격 집필 착수.
- `content/_meta.js`에 `book` 항목, `content/book/_meta.js` 신설(프로젝트 규칙 #13 _meta.js 우선).
- `npm run build` 통과 + `/book` 및 각 장 라우트가 정상 렌더.
- 장 상단 요약 이미지·본문 삽화가 웹에서 반응형으로 표시(`public/images/book/` 경로 확인).
- `index.md` 상단 PDF 다운로드 버튼이 `/book/promised-data.pdf`를 정상 링크.
- 사이드바에 부/장 순서가 목차대로 노출.
- `/book`이 로그인 없이 접근 가능(공개), 기존 `/til`·`/study` 게이트는 그대로 유지.

## 명시적으로 안 하는 것 (YAGNI)

- **PDF 자동 생성 파이프라인** — `/book/full` 통합 뷰 + 브라우저 인쇄로 충분. 서버측
  PDF 렌더(Pandoc/Playwright) 도입은 안 한다.
- **데모 저장소** — 위 "이력서 보강"은 선택 stretch, 이번 코어 범위 밖.
- **삽화 자동 생성** — 프롬프트만 제공, 생성은 사용자가 외부 툴로.
- **인증/유료화** — 외부 공개, 단일 배포.
- **분산 트랜잭션 심화(별도 부)** — 18장 1개로 개괄.
- **댓글·조회수 등 인터랙션 기능** — 정적 읽기 전용.

## 열린 질문 (구현 계획에서 확정)

- 스타일 가이드의 구체 화풍(등장인물 디자인)은 1장 착수 시 사용자와 함께 확정.
- `content/book/`이 메인 위키 사이드바/MOC에 어떻게 노출될지(별도 top-nav 항목 vs 그룹핑) —
  의도된 노출 방식 확인.
